Cross-wave summary of the most serious findings:

1. **Stdlib never type-checks.** `fuse build DIR/...` (the canonical
   library-mode command used by W20/W21/W22's Verify line) only calls
   `parse.Parse` — no resolve, no check, no consteval. The whole
   `stdlib/core/` + `stdlib/full/` surface ships without the checker
   ever looking at a body. Combined with the fact that every non-trivial
   stdlib method body is literally `return 0;` / `return true;`, this is
   a live reproduction of [L002](../learning-log.md#L002) — the exact
   failure mode the rules were written to prevent.
2. **W17 codegen helpers are orphaned.** `EmitReprAttr`, `EmitAlignAttr`,
   `EmitInlineAttr`, `EmitIntrinsic`, `EmitVariadicCall`, `EmitPtrNull`,
   `EmitSizeOf`, `EmitAlignOf`, `EmitSizeOfVal`, `EmitUnionLayout`,
   `EmitOverflowAdd/Sub/Mul`, `EmitLocalName`, `EmitDebugInfoBlock`
   all exist, all have passing unit tests, and are **never called from
   `compiler/codegen/c11.go` or any other package**. W17 phases 05, 06,
   07, 08, 09, 10, and 12 are ceremonial. This is the textbook
   [L013](../learning-log.md#L013) pattern.
3. **W16 runtime proofs do not exercise surface Fuse.** Both
   `TestSpawnObservable` and `TestChannelRoundTrip` construct MIR
   directly and bypass the front end. The `.fuse` files are
   commentary — the runtime never sees compiled source.
4. **W21 allocator proof is Go-simulated.** `TestBumpAllocatorProof`
   runs a hand-written Go bump-arena; no Fuse code is compiled. The
   `bump_allocator.fuse` file is a conceptual comment-only placeholder.
5. **W23 two-crate proof does not use the dependency.** `root/src/main.fuse`
   is `fn main() -> I32 { return 42; }`. The dependency `mathlib::answer`
   is never called. The manifest parses; the lockfile writes; nothing
   computed by the dependency reaches `main`.
6. **W18 incremental cache is never wired into the driver.** No symbol in
   `compiler/driver/build.go` references `Cache`, `PlanIncremental`, or
   any fingerprint helper. `TestIncrementalEditCycle` asserts only
   *"warm build ≤ 4× cold build"* — a bound that any deterministic
   compiler satisfies regardless of caching.
7. **W19 LSP never runs the checker.** `computeDiagnostics` invokes only
   `parse.Parse`. Hover, goto-definition, completion, and symbols all
   route through `doc.Extract` (line-based doc-comment scan) rather than
   the resolver or type table. Type errors never reach the editor.
8. **@global_allocator is documentation only.** `@global_allocator`
   appears in the W21 wave doc and in `stdlib/core/alloc/global.fuse` as
   a comment, but the string is not present anywhere under `compiler/`.
   Nothing recognises the attribute.
9. **Perf baseline measures nothing about the compiler.** The W17
   "compiler self-host proxy" benchmark (`BenchmarkLexParseCorpus`)
   never calls `lex.Scan` or `parse.Parse` — it counts whitespace with
   `countWordsInBytes`. `BenchmarkMonomorphHeavy` is `sum += j*(j+1)`.
   `BenchmarkChanSpawnRT` uses Go channels. All three have code
   comments admitting the W25/W27 plan is to *replace* them with
   real measurements.

  const_fn, etc.) remain green.
- C — pass: the invariants [L003](../learning-log.md#L003),
  [L005](../learning-log.md#L005), [L006](../learning-log.md#L006) are
  now encoded in `CheckNoMoveAfterMove`, `mir.Function.Validate`, and
  the structural-divergence pass. Unit tests cover the contract.
- D — pass with concession: `D4` (pointer categories) is captured in
  `EmitPointerCategory`, but see §W17 — the helper is not wired, so the
  contract is invariant-locked at the MIR level, not at emission.
- E — pass: the wave is scoped to MIR shape.
- F — pass.
- G — concessions visible in the closure: `CastPtrToInt` /
  `CastIntToPtr` enum constants exist but the classifier rejects them
  ("`W15`'s scope is limited to the scalar ladder"). This is **not** an
  active stub in `STUBS.md` — the rejection is silent from the user's
  perspective. Rule 6.9 requires a diagnostic.

Red flags hit:

- `mir.CastMode` declares `CastPtrToInt` / `CastIntToPtr`, but
  `classifyCast` returns `CastInvalid` for those inputs and there is no
  row in `STUBS.md` Active. Any user program that writes `ptr as
  U64` / `n as *U8` hits a generic-rejection path without the declared
  diagnostic text. (WC015 explicitly notes the gap.)
- The "seal invariant" is admitted to be "detectable but not enforced"
  (WC015). The absence of a central `EmitAfterSeal` gate means a future
  lowering pass can produce post-terminator emissions and only the
  validator catches them at Validate time — too late for a diagnostic
  that points at the offending source span.
- `!=` lowers as `OpSub(1, eq)` rather than through a dedicated
  `OpNot`. Codegen renders this as `r_neg = (int64_t)1 - r_eq;` and
  relies on the preceding op producing exactly 0 or 1. A future op
  that yields `{0,1,-1}` or similar would silently break polarity. The
  contract is locked by one unit test, not by MIR invariant.

Verdict: **needs-rework**.

Smallest corrective action: add a row in `STUBS.md` for pointer-cast
lowering with the diagnostic the classifier must emit and a concrete
retiring wave (W16 or W17), then emit that diagnostic when the
classifier rejects a pointer cast.

Checklist results:

Verdict: **incomplete (major)**.

## W19 — Language Server

Closure SHA: not identified in this audit.

Checklist results:

- A — pass with limitation.
- **B — fail.** The proof program is `TestLspScriptedSession`, which
  launches the server and asserts that initialize / hover / goto /
  completion / documentSymbol all *reply*. It does not assert that the
  replies are semantically correct — it never injects a syntactic
  error, checks that the error shows up as a diagnostic, or confirms
  that hover returns a *type*. Rule 6.10 says exit criteria include
  behavioral requirements; the scripted session tests protocol
  roundtrip, not behavior. A server that answered every hover with
  `"unknown"` and every goto with "nowhere" would still pass.
- **C — fail.** [L007](../learning-log.md#L007) (pattern matching) is
  not the relevant trap here, but the L013 pattern is: the LSP never
  runs resolve, check, or consteval.
  [compiler/lsp/diagnostics.go:19](../../compiler/lsp/diagnostics.go)
  calls only `parse.Parse`. Type errors never reach the editor.
  [compiler/lsp/handlers.go:209,240,276,306](../../compiler/lsp/handlers.go)
  route hover / goto / completion / symbols through `doc.Extract` (a
  line-based doc-comment scan), not through the resolver or the
  type-table. Exit criterion "hover returns a resolved type" is
  literally unsatisfiable on this code path — there is no resolver in
  the call chain.
- D — pass for what it implements.
- **E — fail.** LSP is a cross-cutting feature (parse, resolve, check,
  incremental, transport, wire format). It is a single wave — good —
  but its integration points do **not** each have a proof. The wave has
  one scripted session; there is no proof for "hover returns a
  resolved type", no proof for "goto-definition works for items /
  trait methods / imported names", no proof for "completion offers
  identifiers in scope with type hints".
- F — pass for the minimum diagnostic shape.
- **G — fail.**
  - `Position.character` is treated as bytes, not UTF-16 code units
    (WC019 admits this). This violates the LSP 3.17 spec. Any non-ASCII
    Fuse source produces off-by-N column positions in the editor.
  - Hover / goto / completion / symbols work only single-document.
    Cross-file navigation is deferred to W23 (per WC019) — but
    W23 did not deliver it, and no `STUBS.md` row tracks it.
  - The `computeDiagnostics` comment claims it runs "parse + resolve
    at W19", but the code only calls `parse.Parse`. Comment-vs-code
    mismatch.
  - Hint-field surfacing through LSP Diagnostic is "currently dropped"
    per WC019. That breaks Rule 6.17 passthrough — the editor receives
    diagnostics without the suggestion the user is supposed to see.

Red flags hit:

- Exit criteria declare semantic capabilities that the code cannot
  deliver (resolver is not called).
- Scripted session tests protocol, not behavior — direct L013 shape.
- Several workarounds are noted in the closure without corresponding
  `STUBS.md` entries: byte-vs-UTF-16 positions, doc.Extract-backed
  navigation, regex-based semantic tokens, dropped hint field.

Verdict: **needs-rework**.

Smallest corrective actions:

1. Run resolve + check inside `computeDiagnostics` so type errors
   stream to the editor.
2. Route hover through the real type table and goto through the real
   resolver.
3. Either fix `Position.character` to UTF-16 code units or add an
   Active `STUBS.md` row naming the diagnostic for non-ASCII sources
   and a retiring wave.
4. Wire Rule 6.17 Hints into the LSP Diagnostic payload.

## Cross-wave structural findings

These findings cross wave boundaries and surface only when several
waves are audited together.

### F1. The `fuse build DIR/...` library-mode path is parse-only

[cmd/fuse/main.go:180–230](../../cmd/fuse/main.go) shows `runBuildLib`
calling `parse.Parse` and nothing else. Every wave that relies on
"`fuse build stdlib/core/...` returning N files ok (lib mode)" as
part of its Verify line (W20, W21, W22) is measuring parser success,
not stdlib correctness. This converts all three waves' closures into
[L002](../learning-log.md#L002) violations by default.

### F2. Placeholder bodies are the dominant stdlib shape

`grep -R 'return (0|true);' stdlib/` shows dozens of matches across
stdlib/full/{time,thread,chan,os,fs}.fuse and a smaller number in
stdlib/core. Combined with F1, **no stdlib method has been executed by
a compiled Fuse binary.** This directly collides with working-principle
#4: "Stdlib is the compiler's semantic stress test."

### F3. Proofs-as-Go-simulations cluster in W16, W21, W22

Three waves use Go-side simulation tests as their proof programs:

- W16: spawn / channel proofs construct MIR directly, not through the
  front end.
- W21: bump-allocator proof is a Go `bumpArena` struct with a Go `bumpVec`
  struct; the `.fuse` file is comments only.
- W22: reuses the W16 MIR-constructed proofs.

Each closure explicitly invokes parser / checker / lowerer gaps as the
reason. Without `STUBS.md` rows naming each gap and a retiring wave, a
future auditor cannot tell which of these are tracked work and which
are unreported deferrals.

### F4. Cross-cutting features lack per-seam proofs

LSP (W19), incremental driver (W18-P04), stdlib core (W20), stdlib
hosted (W22), and cross-crate `use` (W23) all span multiple pipeline
seams. Each ships with fewer than one end-to-end source-compiling
proof per seam. [L015](../learning-log.md#L015) is the carried-forward
rule against exactly this pattern.

### F5. Wave-deferral references are drifting out of the plan

Several closures defer work to waves whose plan does not schedule that
work:

- WC022 refers to "W24 macro-dispatch wave" — W24 is the Stub
  Clearance Gate.
- WC023 defers `fuse vendor` to "W24's job" — same.
- WC023 defers cross-crate `use` to "W25" — W25's plan lists
  self-hosting, not cross-crate resolution.
- WC021 defers `@global_allocator` to "W23 package management" — W23
  did not touch it.

This is a slow-motion integrity drift: each deferral is plausible in
isolation, but the aggregate effect is that a real feature vanishes
into a wave that never plans to land it.

### F6. Active stubs table is "empty" but the codebase contains many live gaps

`STUBS.md` Active table currently holds only the remaining project-
level rows (W24 gate, W25 self-hosting, W26 native backend, W27 perf
gate, W28 retirement, W29 target matrix, W30 docs). Every other row
has been retired. Yet:

- Pointer-cast lowering is silently rejected (W15).
- `ThreadError` result wrapping is unimplemented (W16).
- Capturing-closure spawn lifts are unimplemented (W17).
- `@repr`, `@align`, `@inline`, `@cold`, intrinsics, variadic,
  `Ptr.null`, `size_of`/`align_of`, `size_of_val`, union-layout,
  overflow policy emission are all orphaned helpers (W17).
- Debug-info breakpoint contract is unexercised on Windows (W17).
- Perf benchmarks are Go stand-ins (W17).
- Incremental cache is not wired to the driver (W18).
- LSP resolver / checker / UTF-16 positions / hint passthrough /
  cross-file navigation / typed hover / typed goto / scope-aware
  completion are all partial or absent (W19).
- Stdlib bodies are placeholders (W20, W21, W22).
- `@global_allocator` recognition is docs-only (W21).
- `fuse vendor` is a placeholder (W23).
- Cross-crate `use` is unimplemented (W23).

Rule 6.15 (overdue-stub check) and Rule 6.9 (silent-stub prohibition)
are both systemically at risk. The empty Active table does not reflect
reality. Before **W24 Stub Clearance Gate** can begin, every one of
the above needs a row with a diagnostic, a file:line, and a retiring
wave — or explicit evidence that the behavior is now correct.

### Fix 3 — W20 doc.CheckMissingDocs scope inheritance

**Audit findings addressed:** W20 G-item (WC020: "`doc.CheckMissingDocs`

private.

### Fix 5 — F1: `fuse build DIR/...` promoted to parse+resolve+check

**Finding addressed:** W18 F1 — `runBuildLib` in

**Change:**
1. `cmd/fuse/main.go` — `runBuildLib` now threads every
   `.fuse` file through `parse → resolve → hir.Bridge → check`
   and fails fast with a non-zero exit on the first file that
   surfaces diagnostics. Imports (`check`, `hir`, `resolve`,
   `typetable`) were added.
2. Promoting this surfaced two latent bugs that the unit tests
   hadn't caught because the prior test corpus never typed
   `self`/`Self` inside an impl or trait body:

   path-lookup branch) is populated, and a residual `Infer`
   param inside an impl/trait context falls back to `selfT`
   rather than diagnosing.

TypeId is correct by the time MIR lowering and W17 codegen
consume it — those layers don't have to reinvent Self
substitution. The checker's `scope.selfT` is kept in sync so
the `expr.go` `self` branch still functions (belt + braces),
and the `Infer → selfT` fallback guards against a future
AST corner the bridge might miss. The selfType field is
stack-push/pop discipline (save/restore) so nested impls
(reserved for later waves) don't leak across frames.

### Fix 6 — W19 LSP computeDiagnostics + hint passthrough

**Finding addressed:** W19 F1 — the LSP's `computeDiagnostics`
ran **parse only** and silently returned after producing the
lexer / parser diagnostic list. Every resolve error, every
type-check error, every checker hint flowed to `fuse build`
but was **invisible in the editor**. The LSP Diagnostic
translator also dropped the `lex.Diagnostic.Hint` field, so
quick-fixes could never fire on server-originated
diagnostics (the quick-fix extractor at `handlers.go:150`
searches the Message for ` hint: `).

**Change:**
1. `compiler/lsp/diagnostics.go` — `computeDiagnostics` now
   threads the document through `parse → resolve →
   hir.Bridge → check`, concatenating each pass's
   diagnostics. When parse returns a nil AST the pipeline
   short-circuits (no point running resolve on an unparseable
   file); otherwise subsequent passes run even when earlier
   passes produced soft errors, so a single keystroke
   surfaces as many actionable diagnostics as possible.
2. A `runDownstreamPasses` helper runs resolve/bridge/check
   with a `recover()` guard. The checker expects well-formed
   programs and can panic on partial input mid-keystroke;
   recovery keeps the server alive and bounds the blast
   radius to "this tick produced no check diagnostics".
3. `translateDiagnostics` now appends
   `lex.Diagnostic.Hint` to `Message` as ` hint: <text>`
   when present. That matches the convention the checker
   already uses for its own diagnostics and is exactly the
   shape the code-action extractor reads back, so
   quick-fixes work end-to-end.

**Rationale (robustness):** The LSP and CLI now run the same
pipeline — any drift in semantics between `fuse build` and
the editor becomes immediately visible. Guarding only the
downstream passes (not parse) keeps the fastest path
(syntax error) unguarded for transparency while the more
fragile semantic passes are isolated behind recover(). The
hint passthrough rides the existing Message-suffix
convention rather than changing the LSP Diagnostic schema,
so no client-side changes are needed.

**Test (regression):**
- `compiler/lsp/lsp_test.go` — `TestLspDiagnosticsSemantic`
  opens a document that parses cleanly but fails type-check
  (returns `Bool` where `I32` is declared) and asserts the
  published diagnostic carries an inline ` hint: ` payload.
  This pins BOTH the pipeline promotion AND the hint
  translation.

**Verification:**
```
go build ./...                                                      # clean
go test ./compiler/lsp/ -run TestLspDiagnosticsSemantic -v          # PASS
go test ./compiler/lsp/...                                          # all PASS (incl. existing stream + quick-fix tests)
```

**Residual:** The diagnostic stream is computed per-document,
treating every file as its own compilation unit. A
cross-module surface (e.g. `use` of another file's symbol)
will resolve against an empty world and may produce an
"unresolved" flag that `fuse build` doesn't produce.
Multi-file LSP awareness is out of scope for this audit fix;
the tradeoff buys parity on single-file diagnostics today.

### Fix 7 — W18 driver.Cache wired into driver.Build

**Finding addressed:** W18 F2 — `driver.Cache` existed as a
fully-tested on-disk pass-output store, but `driver.Build`
never consulted it. The audit called this a Rule 6.9 silent
stub: the W18 wave closed with a green cache module yet
zero end-to-end incrementality.

**Change:**
1. `compiler/driver/build.go` — `BuildOptions` grows an
   optional `Cache *Cache` field. When set, `Build` computes a
   content-addressed key over:
   - source bytes (parse / resolve / check / codegen all
     depend on these),
   - `Source` / `Output` paths,
   - `Debug` and `KeepC` flags (both influence codegen and
     host-compiler invocation),
   - detected host C compiler identity (`cc.Compiler.Path` +
     `Kind`).
2. On a hit the recorded `BuildResult` JSON is decoded and
   returned — BUT only after `os.Stat`'ing `BinaryPath`. A
   missing or unreadable binary degrades to a miss, so the
   cache cannot silently return a phantom path.
3. On success `Build` encodes the `BuildResult` and calls
   `Cache.Put(key, "build", buildCacheSchema, payload)` +
   `Cache.Flush()`. The schema version `build-v1` will be
   bumped if the payload shape ever changes, invalidating
   older caches.
4. `BuildOptions.Cache` defaults to nil; pre-existing callers
   (every test, every CLI subcommand that doesn't want
   caching) stay on the miss-always path and pay nothing.

**Rationale (robustness):** Binding the key to the
host-compiler identity prevents the common footgun of a
cache surviving a `gcc → clang` switch. Degrading to a miss
on a missing binary is the only safe story — Users routinely
run `rm -rf build/` or `git clean -fdx` without understanding
the cache, and returning a path to a deleted file is worse
than recomputing. Keeping the Cache opt-in via
`BuildOptions.Cache` means the pkg / stdlib integration
tests don't inherit implicit global state, which is the
single most common source of flakes in incremental-build
systems.

**Test (regression):**
- `compiler/driver/build_test.go` —
  `TestBuildUsesCacheOnSecondInvocation`:
  1. First Build populates the cache (assert `Size() ≥ 1`).
  2. Second Build with the same source must return the same
     `BinaryPath` AND leave the binary's mtime unchanged —
     a recompile would rewrite the file.
  3. Removing the binary and calling Build a third time must
     re-run the pipeline (verified by `os.Stat` on the new
     binary) rather than returning a phantom `BuildResult`.

**Verification:**
```
go build ./...                                                           # clean
go test ./compiler/driver/ -run TestBuildUsesCacheOnSecondInvocation -v  # PASS
go test ./compiler/driver/...                                            # all PASS (incl. existing cache, incremental, pkg, specialize tests)
```

**Residual:** The cache key hashes the **top-level** source
file plus the tuning knobs; it does not descend into `use`d
files, so a change to a transitive import would not
invalidate the key today. The one-file crate W05 driver
doesn't have a notion of transitive imports yet, but this
is a real gap to close when multi-crate builds arrive
(W23-era). Also — the cache does not evict old entries;
size bounds / LRU are a later concern.

### Fix 8 — W23 two-crate proof actually uses the dep

**Finding addressed:** W23 F3 — `tests/e2e/two_crate_project_test.go`
wrote a `fuse.lock` and compiled the root crate but never
verified that the declared path-dependency (`mathlib`) flowed
through the resolver into the lockfile's `Entries`. The test
was W23's verify target; its silence meant a resolver bug
that dropped deps would pass CI unnoticed.

Digging revealed a **real bug** hidden by the weak assertion:
`pkg.Resolver.initialConstraints` at `compiler/pkg/resolver.go:205-207`
explicitly skips dependencies whose `Version` field is empty
("path/URL deps bypass the resolver") — so path-only manifests
produce an empty `Resolution.Entries` and an empty lockfile.
The audit's "test doesn't use the dep" was actually masking
"lockfile doesn't record the dep at all".

**Change:**
1. `compiler/driver/pkg_integration.go` — after
   `Resolver.Resolve` (and in the `hasOnlyPathDeps` fallback
   branch), a new `appendPathDeps` helper walks
   `Manifest.Dependencies`, appends one `LockedCrate` per
   path-dep to the lockfile, and re-finalises. The helper:
   - Resolves the declared path against the manifest's
     directory so the `Source` column records an absolute
     path usable from any working directory.
   - Reads the dep crate's own `fuse.toml` to populate
     `Version` with the crate's declared version; falls back
     to the sentinel `0.0.0-path` when the file is missing
     or malformed so downstream consumers can still
     distinguish path deps from registry ones.
2. `tests/e2e/two_crate_project_test.go` — Contract 3a added:
   - Asserts `lk.Entries` contains a `mathlib` entry with a
     non-empty `Version` and `Source`.
   - Asserts the recorded `Source` directory contains a
     readable `src/lib.fuse`.
   - Asserts `lk.Digest` is non-empty post-`Finalize` (Rule
     7.1 byte-stability contract).

**Rationale (robustness):** Path deps are the W23-MVP surface;
the resolver pipe-cleaning them out of the lockfile silently
breaks every downstream consumer (fetcher, build-graph
assembly, reproducible-build guarantees). Fixing at the
driver layer keeps `pkg.Resolver` pure (no filesystem
access) while still producing an accurate lockfile. Reading
the dep's own manifest for its version is a small I/O cost
paid once per resolution and materially improves the
lockfile's usefulness; the `0.0.0-path` sentinel avoids
lying when the dep manifest can't be read. The test now
pins all three: the entry exists, it points at a real
location, and the lockfile is byte-stable. Self-hosted
cross-crate `use` (actually calling `mathlib::answer()`
from `main.fuse`) remains W25 work — STUBS.md already
carries the Active "cross-crate use (W25)" row.

**Test (regression):**
- `tests/e2e/two_crate_project_test.go` —
  `TestTwoCrateProject` now fails if any link in the chain
  breaks (manifest parse → resolver → path-dep augmentation
  → Finalize digest → on-disk lib source reachable).

**Verification:**
```
go build ./...                                                  # clean
go test ./tests/e2e/ -run TestTwoCrateProject -v                # PASS (includes the new Contract 3a)
go test ./compiler/pkg/... ./compiler/driver/... ./tests/e2e/...  # all PASS
```

**Residual:** Transitive path deps (a path dep whose own
manifest declares another path dep) would need the augmenter
to recurse. The current implementation appends only the
root manifest's direct path deps; this matches the W23
scope (one-level path deps are the proof target) but is a
known limitation to revisit with W24 workspaces or W25
cross-crate resolution, whichever lands first.

### Fix 9 — W17 smallest emitters wired into the main codegen path

**Finding addressed:** W17 F2 — `EmitPtrNull`, `EmitSizeOf`, and
`EmitIntrinsic` existed in
`compiler/codegen/w17_intrinsics.go` and passed their own unit
tests but were **never reachable** from `EmitC11`. The main
codegen dispatch had no MIR op and no OpCall callee-name path
that routed to any of them. Rule 6.9: silent stubs.

**Change:**
1. `compiler/codegen/c11.go` — `TermUnreachable` now emits
   `__builtin_unreachable()` (via `EmitIntrinsic("unreachable", nil)`)
   in addition to the existing `fuse_rt_abort("unreachable")`
   guard. The intrinsic is the optimizer hint; the abort is
   the belt-and-braces trap if the intrinsic folds away in a
   release build. This is a real end-to-end wire: every
   unreachable block in every Fuse program now flows through
   `EmitIntrinsic`.
2. `compiler/codegen/c11.go` — `OpCall` now consults
   `tryEmitCompilerIntrinsic(in)` **before** emitting the
   default `name(args...)` path. A callee whose name starts
   with the sentinel prefix `__fuse_intrinsic_` is routed to
   the intrinsic helpers:
   - `__fuse_intrinsic_unreachable` / `likely` / `unlikely` /
     `assume` / `fence` / `prefetch` → `EmitIntrinsic`.
   - `__fuse_intrinsic_ptr_null__<targetType>` → `EmitPtrNull`
     with the target type decoded from the `__<payload>` suffix.
   - `__fuse_intrinsic_size_of__<bytes>` → `EmitSizeOf` with
     the byte count decoded from the suffix.
3. `compiler/codegen/w17_intrinsics.go` — new
   `tryEmitCompilerIntrinsic` helper plus a `cutOnce` shim for
   `strings.Cut`. Both are deliberately small and total: any
   unrecognised name returns `(ok=false)` so the OpCall caller
   falls through to the plain `name(args...)` emission.

**Rationale (robustness):** Adding MIR opcodes would have been
a wider change (mir pkg + every walker / test that enumerates
ops). Routing by callee-name prefix stays surgical: the MIR
surface is unchanged, the lowerer can start producing
intrinsic calls at any future point, and every non-intrinsic
OpCall still takes the default path. The sentinel prefix
(`__fuse_intrinsic_`) is user-hostile as a real C identifier,
so the risk of name collision with a legitimate Fuse fn is
zero. `TermUnreachable` was the natural first real wire:
every unreachable already emits a codegen line, so there was
no source-level plumbing to invent.

**Test (regression):**
- `compiler/codegen/w17_test.go` — `TestStructuralDivergence`
  now asserts `__builtin_unreachable()` appears in the
  emitted C (alongside the pre-existing `fuse_rt_abort`
  assertion).
- `compiler/codegen/w17_test.go` — new
  `TestCompilerIntrinsicDispatch` sub-tests drive
  `EmitC11` with three synthetic OpCalls
  (`__fuse_intrinsic_unreachable`,
  `__fuse_intrinsic_ptr_null__int64_t`,
  `__fuse_intrinsic_size_of__16`) and assert the output
  contains the expected intrinsic expansion AND does not
  contain the unresolved `__fuse_intrinsic_*(` call site.

**Verification:**
```

go test ./compiler/codegen/...                                                                  # all PASS
```

**Residual:** The lowerer does not yet emit
